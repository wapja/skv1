skv1 starter kit — handleiding

Een nieuw project bootstrappen vanuit de kit. Werkt op elke macOS-machine met Herd Pro. Versie v0.1.4 · 2026-05-08

Wat is skv1?

skv1 is een Laravel 13-startkit voor multi-tenant invitation-only apps: subdomein-tenants, RBAC met spatie/laravel-permission, invitations met signed activation links, impersonation, audit log, health-check, backups. De repo wapja/skv1 is gemarkeerd als GitHub Template — daar bouw je nieuwe projecten op.

Voorwaarden per machine — eenmalig

Doe dit één keer per machine waar je skv1-projecten wilt draaien. Op je hoofdmachine staat dit waarschijnlijk al; op een nieuwe machine moet je het inrichten.

1. Laravel Herd Pro

Download en installeer Herd Pro: herd.laravel.com. Brengt PHP, Nginx, Postgres, Mailpit, Composer, Node ineens mee.

2. Flux Pro license in ~/.composer/auth.json

De kit gebruikt Flux UI Pro. Composer heeft daarvoor http-basic auth nodig. Maak het bestand aan (mode 600):

mkdir -p ~/.composer
cat > ~/.composer/auth.json <<'EOF'
{
    "http-basic": {
        "composer.fluxui.dev": {
            "username": "<jouw e-mailadres>",
            "password": "<jouw flux pro license key>"
        }
    }
}
EOF
chmod 600 ~/.composer/auth.json

Of, sneller, via Composer zelf:

composer config --global http-basic.composer.fluxui.dev <email> <license-key>

Op een nieuwe machine kan je het bestand ook gewoon van je hoofdmachine kopiëren:

scp ~/.composer/auth.json <nieuwe-machine>:~/.composer/auth.json

3. SSH-key voor GitHub

De kit-repo wordt via SSH gekloond. Op een nieuwe machine genereer je een key en hang je de public-key in je GitHub-account onder Settings → SSH and GPG keys:

ssh-keygen -t ed25519 -C "<jouw e-mailadres>"
cat ~/.ssh/id_ed25519.pub  # kopieer naar GitHub

Test:

ssh -T git@github.com
# verwacht: "Hi <username>! You've successfully authenticated, ..."

Per nieuw project — overzicht

Een nieuw project opzetten kost twee stappen op GitHub en één commando-blok in je terminal:

  1. Op GitHub: "Use this template" klikken → nieuwe repo aanmaken.
  2. Op je machine: kloon, bootstrap, herd link, DB aanmaken, migrate, build.

Per nieuw project — stappen

1Maak een nieuw GitHub-repo van het sjabloon

  1. Open github.com/wapja/skv1.
  2. Klik de groene knop Use this template → Create a new repository.
  3. Kies Owner en Repository name (bv. webedu). Visibility naar smaak.
  4. Klik Create repository. GitHub maakt een schone kopie zonder skv1's commit-historie of tags.

2Lokaal opzetten (TL;DR)

Eén commando-blok. Vervang webedu door je projectnaam.

# vervang webedu door je projectnaam
PROJECT=webedu
GITHUB_USER=wapja  # of jouw GitHub-account/organisatie

cd ~
git clone git@github.com:${GITHUB_USER}/${PROJECT}.git
cd ${PROJECT}

composer install
mkdir -p storage/framework/{cache,sessions,views,testing} storage/logs

cp .env.example .env
sed -i '' "s/skv1/${PROJECT}/g" .env phpunit.xml
php artisan key:generate

herd link ${PROJECT}
herd secure ${PROJECT}

PSQL=~/Library/Application\ Support/Herd/bin/psql
"${PSQL}" -U postgres -h 127.0.0.1 -c "CREATE DATABASE ${PROJECT};"
"${PSQL}" -U postgres -h 127.0.0.1 -c "CREATE DATABASE ${PROJECT}_test;"

php artisan migrate --seed
npm install && npm run build
php artisan optimize:clear

echo "→ open https://demo1.${PROJECT}.test/login"
Werkt dit op een verse machine niet? Loop de drie voorwaarden hierboven na. Veelal zit het in een ontbrekende Flux Pro auth.json (composer install crasht) of een onbekende SSH-key (git clone faalt).

3Inloggen en verifiëren

Open in je browser:

https://demo2.${PROJECT}.test/login admin@demo2.local / Password123! Tenant demo2, organisation_admin

4De kit-restanten opruimen (optioneel)

Voor een schone start van je eigen project verwijder je de docs die alleen voor de kit-historie relevant zijn:

cd ~/${PROJECT}
rm OUTPUT_SUMMARY.md CHANGELOG.md
rm -rf docs/superpowers/

# README.md aanpassen aan je eigen project
# eventuele eigen demo-data: database/seeders/Demo*Seeder.php aanpassen of weghalen

git add -A
git commit -m "chore: rename skv1 → ${PROJECT}, drop kit-only docs"
git push

Veelvoorkomende problemen

Mail wordt niet verstuurd / queue-job faalt met "Connection refused" op 127.0.0.1:1025

De Herd Pro mail-service luistert op poort 2525, niet op de Mailpit-default 1025. .env.example sinds v0.1.4 staat correct ingesteld. Voor projecten gegenereerd vóór v0.1.4: zet in .env MAIL_PORT=2525, MAIL_USERNAME="${APP_NAME}" en MAIL_ENCRYPTION=null. Zorg ook dat de mail-service aanstaat in de Herd-app (Services → Mail) — controleer met lsof -i :2525.

Inloggen geeft HTTP 500 op /dashboard (lege response)

Bug in het BelongsToOrganisation global scope: een recursieve auth()->user() tijdens session-load → C-stack overflow → PHP-FPM segfault. Opgelost in v0.1.2. Voor projecten die vóór v0.1.2 zijn aangemaakt: vervang in app/Models/Concerns/BelongsToOrganisation.php de regel $user = auth()->user(); binnen de global scope door $user = auth()->hasUser() ? auth()->user() : null;.

composer install crasht met "the requested package is not available"

Flux Pro auth ontbreekt of is verkeerd. Controleer ~/.composer/auth.json. Test met:

composer config --global --list | grep flux

git clone vraagt om wachtwoord of geeft "Permission denied"

Je SSH-key is niet geregistreerd bij GitHub of de juiste key wordt niet aangeboden. Check ssh -T git@github.com.

php artisan crasht met "Please provide a valid cache path"

De storage-mappen ontbreken. Run het mkdir storage/framework/{...} commando uit stap 2.

https://demo1.${PROJECT}.test geeft een browser-waarschuwing

Eerste keer na herd secure: hard refresh (cmd+shift+R). Werkt het dan nog niet, run herd restart.

Site verschijnt niet in Herd Pro's GUI

De CLI (herd links) is autoritair en toont hem wel. De GUI cached zijn list — klik een ander tab en terug, of herd restart.

Adminer geeft "Cannot redeclare class Min_DB"

OPcache-glitch in Herd's Adminer. Fix: herd restart php + browser hard refresh.

De originele ~/skv1 directory zelf updaten

De kit zelf evolueert. Wil je een bestaand project bijwerken naar een nieuwere kit-versie, dan is dat handwerk: vergelijk de relevante files (vooral app/, config/, migraties, seeders) en cherry-pick. Een verse nieuw project is altijd op de laatste kit-versie omdat "Use this template" altijd de huidige main-staat pakt.
